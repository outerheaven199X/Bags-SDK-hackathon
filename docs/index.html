BagsSDK

An MCP server for the Bags.fm API. 40 tools, 4 browsable resources, 6 workflow prompts, a fee-split builder, and an optional autonomous agent. One install.

40 MCP tools
Trading, launches, fee sharing, partner keys, Dexscreener, agent auth. If Bags has an endpoint, there's a tool for it.
4 resources
Launch feed, pools, token detail, portfolio. Data your AI can read before it decides what to do.
6 prompts
Multi-step recipes for launching tokens, claiming fees, checking earnings. The AI follows the steps, you sign the transactions.
Fee config composer
100-party fee splits without raw BPS math. Fluent API, validation, presets for common patterns.
Zero custody
No private keys touch this server. Transactions come back unsigned. You sign in your own wallet.
Agent mode
Autonomous loop. Hermes 4 for the routine checks, Sonnet for the judgment calls.
Install & run
npx bags-sdk-mcp

What is MCP?

Model Context Protocol is an open standard from Anthropic (also adopted by OpenAI and Google) that lets AI assistants call external APIs. You build one MCP server, and it works with Claude Desktop, Cursor, Windsurf, or anything else that supports the protocol. No custom integrations per client.

BagsSDK is an MCP server for Bags.fm. Connect it to Claude Desktop and you can type "launch a token called XYZ with a 3-way fee split" in plain English. Claude works out which tools to call and in what order.

MCP servers expose three things:

CapabilityWhat it doesBagsSDK count
ToolsActions: get quotes, build transactions, claim fees, launch tokens40
ResourcesRead-only context the AI checks before deciding what to do4
PromptsPre-written scripts for multi-step jobs like token launches6

Quickstart

About 5 minutes. You need a Bags API key and Claude Desktop (or Cursor, or any MCP client).

1. Get your Bags API key

Visit dev.bags.fm, sign in, and create a new API key.

2. Configure Claude Desktop

Open Claude Desktop settings and add this to your MCP server configuration:

claude_desktop_config.json
{
  "mcpServers": {
    "bags-sdk": {
      "command": "npx",
      "args": ["-y", "bags-sdk-mcp"],
      "env": {
        "BAGS_API_KEY": "your_api_key_here"
      }
    }
  }
}

3. Restart Claude Desktop

You should see "bags-sdk" in the MCP servers list. Try asking:

💬 "Show me the latest token launches on Bags"

Claude will call bags_launch_feed and return live data from the Bags.fm API.

4. Try more commands

"What's my SOL balance?" → bags_wallet_balance
"Get a quote to buy 1 SOL of token XYZ" → bags_quote
"Show me my claimable fees" → bags_claimable_positions
"Launch a token called TEST" → bags_launch_token prompt

Resources

Resources are read-only data the AI can look at without making a tool call. When Claude wants to check what tokens exist before deciding what to do, it reads a resource instead of calling bags_launch_feed as a tool. Four are available:

ResourceURITypeCache
Launch Feedbags://launchesStatic5 min
Poolsbags://poolsStatic5 min
Token Detailbags://token/{mint}Template2 min
Portfoliobags://portfolio/{wallet}Template2 min

Template resources take a parameter in the URI (like a token mint or wallet address). The token detail resource pulls pool state, creators, and lifetime fees into one response, which saves the AI three separate tool calls.

Prompts

Prompts are recipes for multi-step operations. Select one and the AI gets step-by-step instructions: which tools to call, in what order, and what to do with the results. You still approve and sign everything.

WORKFLOWbags_launch_token

Solo token launch. Args: name, symbol, description, image URL, creator wallet, initial buy SOL.

Flow: resolve_wallet → compose_fee_config (solo) → create_token_info → create_fee_config → create_launch_tx

WORKFLOWbags_launch_team_token

Team launch with multi-party fee splits. Args include team members as platform:username:percentage.

Flow: resolve_wallet ×N → compose_fee_config → create_token_info → create_fee_config → create_launch_tx → dexscreener_check

WORKFLOWbags_analyze_fees

Fee earnings analysis for a token or wallet. Aggregates lifetime fees, claim events, creators, and stats.

WORKFLOWbags_setup_partner

Create a partner config and understand how to use it. Default 25% fee on tokens launched through your key.

WORKFLOWbags_claim_all

Batch claim all pending fees above a threshold. Returns unsigned transactions for each position.

WORKFLOWbags_portfolio_overview

Full portfolio summary: SOL balance, token holdings, claimable positions, and aggregate claim stats.

Fee Config Composer

Bags.fm supports up to 100 fee claimers per token. The raw API expects arrays of wallet addresses and basis points that need to sum to exactly 10,000. That's tedious and error-prone. The Fee Config Composer handles the math, validates your config, and ships with presets for the patterns people actually use.

ℹ Bags.fm splits 1% of all trading volume among fee claimers. BPS (basis points) must sum to exactly 10,000 (= 100%). Each claimer is identified by a social provider + username.

Fluent API

TypeScript
import { FeeConfigBuilder } from 'bags-sdk-mcp';

const config = FeeConfigBuilder.create()
  .addCreator('twitter', 'nikki', 5000)       // 50%
  .addRecipient('twitter', 'alice', 3000)     // 30%
  .addDividendsBot(2000)                       // 20% to holders
  .validate();
// { valid: true, errors: [] }

Templates

soloCreator100% to one personFeeConfigBuilder.soloCreator('twitter', 'you')

Validation rules

The validator catches the mistakes you'd expect: BPS that don't add up, duplicate recipients, empty arrays, negative values. If you end up with more than 15 claimers, it warns you that Address Lookup Tables are needed (the SDK handles that part).

As an MCP tool

The bags_compose_fee_config tool exposes this as either mode: "custom" (manual recipients) or mode: "template" (use a preset). It validates and returns the config without submitting to the API, so you can review before committing.

Caching & Rate Limits

The Bags.fm API gives you 1,000 requests per hour. That goes fast when an AI agent starts enriching 20 tokens in a single conversation, pulling creator data, fees, and pool state for each one. Five cache tiers, based on how often the data actually changes:

TierTTLWhat's cached
ImmutableForeverSocial → wallet resolution (never changes)
Stable10 minCreators, pool configs, partner configs
Moderate5 minLifetime fees, claim stats, token feed, pools
Volatile2 minClaimable positions
NoneNever cachedQuotes, transactions, auth, send-tx

It's a Map with TTL expiry. Nothing fancy, no Redis, no external deps. Entries expire on read. You can clear a whole category at once with prefix-based invalidation if you need to force-refresh something.

Zero-Custody Design

BagsSDK never stores or handles private keys. Every transaction-generating tool returns an unsigned base64-encoded VersionedTransaction. The user signs externally with their wallet (Phantom, Solflare, Backpack).

The only tool that sends anything on-chain is bags_send_transaction, which requires a pre-signed transaction as input. The signing happens in your wallet, not in BagsSDK.

⚠ Never share your private key with any MCP server. BagsSDK doesn't ask for it and doesn't need it.

Retrieve the live token launch feed. Returns recent and active launches with status, metadata, and creator info.

Input: limit (optional), offset (optional). Cached 5 min.

Step 1 of launching a token. Uploads metadata (name, symbol, description, image) to IPFS via Bags API.

Agent mode

Pass --agent and BagsSDK runs as an autonomous loop instead of waiting for MCP calls. Two LLMs split the work.

How the routing works

Hermes 4 (Nous API, fast, cheap) does the repetitive stuff: checking positions, claiming fees when they cross a threshold, watching the launch feed for new tokens, pulling portfolio status.

Claude Sonnet (Anthropic API, slower, smarter) gets the judgment calls: is this new launch worth looking at? What's the best fee config for a 5-person team? Where's the money actually going across my positions?

The split is simple: if the task involves "should I," it goes to Sonnet. If it's "do the thing," Hermes handles it.

Strategies

StrategyCLI flagWhat it does
Auto-claim--auto-claimChecks positions every 5 min. Claims anything above 0.01 SOL.
Launch monitor--monitorWatches the feed every 30s. Flags interesting launches. Sends them to Sonnet if they match your criteria.
Portfolio rebalanceCode onlyPulls all your positions, asks Sonnet what to do about them.
Fee optimizerCode onlyReads your fee configs, asks Sonnet if they could be better.
CLI usage
bags-sdk-mcp --agent --auto-claim          # Auto-claim only
bags-sdk-mcp --agent --monitor              # Launch monitor only
bags-sdk-mcp --agent --auto-claim --monitor # Both strategies

Transports

stdio (default)

Works with Claude Desktop, Cursor, and MCP Inspector out of the box. Just run bags-sdk-mcp or npx bags-sdk-mcp.

Streamable HTTP

For remote connections or multi-client setups. Run bags-sdk-mcp --http to start an Express server on port 3000 (or --port=8080 for a custom port).

Two endpoints: POST /mcp for MCP requests and GET /health for uptime checks. Each request is stateless.

Contributing

Adding a new tool

  1. Create a file in the appropriate src/tools/{domain}/ directory
  2. Define a Zod input schema and output schema
  3. Set annotations: readOnlyHint, destructiveHint, idempotentHint, openWorldHint
  4. Implement the handler with try/catch and actionable error messages
  5. Register it in src/tools/_registry.ts
  6. Add a cache tier if the data is stable enough to cache
